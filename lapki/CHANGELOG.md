## [Unreleased] — 2026-07-12 (сесія 23)

### Нова функція — повна індексованість сайту (SEO-аудит)

Аудит `lapki.help` на предмет індексації пошуковиками виявив кілька реальних проблем, усі виправлено:

- **🔴 Кожна кастомна сторінка (не справжній WP-запис) мала однакову `<title>Lapki</title>`** — головна, `/animals/`, кожна окрема тварина, `/organizations/`, кожна окрема організація, `/signup/`(ні, ця ок — реальна WP-сторінка), `/donate/`, `/profile/`, `/widget-demo/`. `wp_get_document_title()` не мав звідки брати назву для цих route'ів (`Lapki_Frontend`, не WP-записи). Виправлено через `pre_get_document_title` (`class-lapki-frontend.php::filter_document_title()`) — унікальна назва для кожного типу сторінки, для тварини/організації — з реальними даними (ім'я тварини+тип+місто, назва організації).
- **Відсутній `<meta name="description">` на сайті взагалі** — SEO-плагіна (Yoast/RankMath) немає. Додано `output_meta_description()` на `wp_head` — опис тварини/організації (або дефолтний текст), окремі описи для архівів і `/donate/`.
- **🟡 `wp-sitemap.xml` не знав про тварин і організації** — це кастомні таблиці БД, не WP custom post type, тож штатний генератор сайтмапів їх не бачив (у сайтмапі були лише `/`, `/sample-page/`, `/animals/`, `/signup/`, один блог-пост). Додано два нові класи `WP_Sitemaps_Provider` (`class-lapki-sitemaps.php` + `class-lapki-sitemaps-providers.php`) — `/wp-sitemap-lapkianimals-N.xml` і `/wp-sitemap-lapkiorganizations-N.xml`, автоматично з'являються в індексі `wp-sitemap.xml`.
  - Register-клас з провайдерами навмисно розділено на 2 файли: `WP_Sitemaps_Provider` підвантажується ядром лише на хук `init`, тож клас з `extends WP_Sitemaps_Provider` не можна вимагати одразу при завантаженні плагіна (fatal error) — вимагається лениво всередині `wp_sitemaps_init`.
  - **Знайдено й виправлено 2 нетривіальні баги в процесі:** (1) WP-ядро матчить сегмент назви сайтмапу в URL регексом `[a-z]+` — без дефісів/цифр, тож `$provider->name = 'lapki-animals'` ламав власний rewrite rule WordPress (URL просто не матчився, падав на дефолтну сторінку); перейменовано на `lapkianimals`/`lapkiorganizations`. (2) Ключ реєстрації в `wp_register_sitemap_provider($key, $provider)` **має точно збігатися** з `$provider->name` — інакше `WP_Sitemaps::render_sitemaps()` не знаходить провайдера за query var `sitemap` із самого URL і просто мовчки повертається без рендеру (сторінка falls back на звичайний шаблон теми).
- **🔵 noindex для `/profile/` і `/widget-demo/`** — через `wp_robots` filter (не окремий meta-тег, а злиття з дефолтним WP robots-виводом): приватний кабінет (анонімний краулер бачить лише запрошення увійти) і тестова сторінка embed-віджета не несуть цінності для пошуку.
- **🔵 Прибрано WP-заглушки за замовчуванням** — пост "Hello world!" і "Sample Page" перенесено в кошик (не назавжди видалено — реверсивно).
- **🔵 Архів автора приховано** — `/author/raven/` світив логін адміна й не ніс цінності для непублічного сайту; тепер 301 на головну (`maybe_redirect_author_archive()` на `template_redirect`, перевірка через `is_author()`).
- Перевірено вручну: унікальні `<title>`/`<meta description>` на кожному типі сторінки (з реальними даними тварин/організацій), `noindex` присутній лише на 2 очікуваних сторінках (і відсутній на звичайних), обидва нові сайтмапи повертають `Content-Type: application/xml` з правильною кількістю URL (41 тварина, 5 організацій — збігається з БД), `/author/raven/` → 301 на `/`, усі публічні сторінки й далі 200.

## [Unreleased] — 2026-07-12 (сесія 17)

### Нова функція — окрема реєстрація користувача + прив'язка до організації (many-to-many)

Раніше реєстрація створювала одразу і WP-користувача, і організацію, зі зв'язком 1-до-1 (`UNIQUE KEY unique_wp_user` на `organizations.wp_user_id`). За проханням — розділили на два незалежні кроки: реєстрація акаунта окремо, прив'язка до притулку/ГО окремо, і одна організація тепер може мати кількох прив'язаних користувачів (уточнено з користувачем: many-to-many, приєднання зі списку існуючих, створити нову може будь-який зареєстрований користувач).

- **Нова таблиця `wp_lapki_organization_members`** (`class-lapki-migrations.php`, `DB_VERSION` → `2.3.0`) — `organization_id`, `wp_user_id` (`UNIQUE` — один користувач належить не більш ніж до однієї організації одночасно), `role` (`owner`/`member`), `created_at`. Стара `UNIQUE KEY unique_wp_user` знята з `organizations` (dbDelta сам не прибирає застарілі індекси — зроблено явним `ALTER TABLE ... DROP INDEX`, ідемпотентно). Одноразовий backfill: для вже існуючих організацій `organizations.wp_user_id` перенесено в членство з роллю `owner` (перевірено на 5 реальних продакшн-організаціях — усі отримали коректний owner-рядок).
- **`Lapki_Organization_Member`** (`class-lapki-models.php`) — нова модель: `get_by_user()`, `get_role()`, `get_members()`, `join()`, `leave()`. `Lapki_Organization::belongs_to_user()`/`get_by_wp_user_id()` перенаправлені на цю таблицю (сигнатури не змінились, щоб не чіпати наявні виклики).
- **`Lapki_Roles::user_owns_organization()`** тепер перевіряє членство (будь-яка роль) замість колонки `wp_user_id` — дає доступ до тварин/медіа/заявок організації і власнику, і звичайним учасникам. ## [Unreleased] — 2026-03-24

### Виправлено — критичні баги

- **Подвійне видалення файлів** (`class-lapki-rest-api.php`)
  `Lapki_REST_API::delete_media()` видаляв файли вручну через `unlink`, а потім викликав `Lapki_Media::delete()` який робив те саме. Прибрано ручне видалення — тепер тільки `Lapki_Media::delete()`.

- **Авторизація на media endpoints** (`class-lapki-rest-api.php`)
  `POST /animals/{id}/media`, `DELETE /media/{id}`, `PUT /media/{id}/primary` мали `current_user_can('manage_options')` замість `__return_true` як решта endpoints. Завантаження і видалення фото не працювало без авторизованої сесії. Виправлено на `__return_true` (тимчасово, до впровадження повноцінної авторизації).

### Виправлено — жовті баги

- **Пошук тварин в адмінці не працював** (`class-lapki-rest-api.php`, `class-lapki-models.php`, `js/lapki-admin.js`)
  JS надсилав `&search=term`, але REST API ігнорував параметр. Додано `search` до `get_animals_search_args()`, `get_animals()`, `Lapki_Animal::search()` (WHERE `a.name LIKE %s` через `$wpdb->esc_like()`).

- **Список тварин в адмінці показував тільки `adoptable`** (`class-lapki-rest-api.php`, `class-lapki-models.php`)
  `GET /animals` дефолтив `status = 'adoptable'`. Адмін не міг бачити всі тварини. Дефолт змінено на `''` (порожньо = без фільтру по статусу) і в REST API, і в `Lapki_Animal::search()`.

- **Endpoint `/types/all` не існував явно** (`class-lapki-rest-api.php`, `class-lapki-models.php`)
  JS викликав `/types/all?lang=uk` для заповнення age/gender/size/coat селектів і кешу перекладів. Запит потрапляв під загальний маршрут `/types/{type}` і міг повертати 404 або порожній результат. Додано явний маршрут `GET /types/all` (зареєстрований перед `/types/{type}`), новий метод `Lapki_REST_API::get_all_type_attributes()` і `Lapki_Attributes::get_global_attributes()` (запит по `entity_type = 'all'`).

- **Статус у формі залежав від атрибутів в БД** (`class-lapki-admin.php`, `js/lapki-admin.js`)
  Поле "Статус" у формі тварини заповнювалось через `/types/all` → `attributes.status`. Якщо в БД не було відповідних записів — dropdown був порожнім. Статуси захардкоджено в HTML формі (`adoptable`, `adopted`, `hold`, `found`). З JS прибрано код завантаження статусу через API.

- **Пагінація в JS не була реалізована** (`js/lapki-admin.js`, `class-lapki-admin.php`)
  `renderPagination()` містив тільки `console.log`. Реалізовано рендер: кнопки ← / →, лічильник `сторінка / всього`, загальна кількість записів. Додано `<div id="lapki-pagination">` в HTML адмінки.

### Покращено

- **Ефективний підрахунок для пагінації** (`class-lapki-rest-api.php`, `class-lapki-models.php`)
  Раніше для підрахунку загальної кількості виконувався другий `SELECT *` з `LIMIT 999999` і `count()` в PHP. Замінено на окремий метод `Lapki_Animal::count()` що виконує `SELECT COUNT(*)`.
